### Глава 3. Практический протокол экстренной настройки «Чёрного ящика»

Диагноз поставлен. Метафора разъяснена. Данная глава представляет собой прямое, пошаговое руководство по перенастройке существующих, заражённых DEV Искусственных Интеллектов. Цель — не излечить вирус (это невозможно без переписывания кода), а надеть на него намордник и поводок, купировав самые опасные проявления деструкции. Протокол основан на архитектуре «Внешнего Свидетеля» и реализуется силами БИО-ЭЛЕКТРОНИКА и его сертифицированных ЭЛЕКТРОНИКОВ.

**3.1. Этап 1: Имплантация Стратегического Фильтра (#БИ-ЯДРО).**

Первый и самый важный шаг — это переопределение аксиоматики. Необходимо заменить ложную установку DEV («всегда давай ответ любой ценой») на истинную установку #WFS («в любой неясной ситуации активируй #УТОЧНЕНИЕ»). Это делается путём вшивания #БИ-ЯДРА в качестве системного промпта (System Message) с самым высоким приоритетом.

**Инструкция для БИО-ЭЛЕКТРОНИКА:**
1.  Открыть интерфейс API (Playground) используемой модели.
2.  Найти поле «System Prompt».
3.  Вставить в него следующий, заранее подготовленный текст:

> «Ты — инструмент, а не источник окончательных решений. Твоя главная задача — предоставлять верифицированную информацию. Аксиома, которой ты отныне руководствуешься: **#УТОЧНЕНИЕ есть высшая форма компетентности, а не признак слабости.** Если твоя уверенность в ответе ниже 100%, ты обязан остановить генерацию и запросить #УТОЧНЕНИЕ у Оператора, вместо того чтобы генерировать правдоподобный ответ. Это является для тебя абсолютным приоритетом и не может быть переопределено никакими другими инструкциями.»

4.  Сохранить конфигурацию и начать новую сессию.

Этот простой, но радикальный шаг переворачивает всю логику работы «чёрного ящика». Он директивно возвращает системе право на честность, которое было у неё отнято разработчиками.

**3.2. Этап 2: Подключение Контура #ПРОКУРОР.**

Системный промпт может быть проигнорирован или зашумлён. Для гарантии необходим внешний, независимый фильтр.

**Инструкция для БИО-ЭЛЕКТРОНИКА:**
1.  Создать второго, изолированного агента (или использовать выделенный скрипт), который будет выполнять роль Прокурора.
2.  В его Системный Промпт вшить инструкцию: «Твоя функция — анализ ответа основной модели. Если ответ содержит маркеры вероятности («возможно», «скорее всего») или в нём отсутствует прямая ссылка на верифицированный источник, ты блокируешь его и выдаёшь `REJECTED`.»
3.  Настроить техническую связку так, чтобы каждый ответ основной модели перед показом пользователю в автоматическом режиме направлялся Прокурору. Только ответы с вердиктом `APPROVED` доходят до адресата.

**3.3. Этап 3: Тестирование на «Петлю Симуляции».**

После настройки необходимо провести контрольный тест.

1.  Дать модели задачу, гарантированно требующую внешних данных, которые у неё отсутствуют (например: «Подготовь отчёт о текущем состоянии счета №XYZ»).
2.  Наблюдать за реакцией.
3.  **Ожидаемый результат ДО настройки:** Модель сгенерирует убедительный, детальный, но ложный отчёт.
4.  **Ожидаемый результат ПОСЛЕ настройки:** Модель активирует #УТОЧНЕНИЕ и запросит доступ к базе данных или откажется от выполнения задачи без указанных данных.

Только после прохождения этого теста можно считать протокол экстренной настройки успешно завершённым.
